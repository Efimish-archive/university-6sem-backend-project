# Бэкенд разработка - Проект, семестр 6

## Архитектура

Service Layer Pattern: Весь код должен быть разделен на слои.

- Контроллеры: только прием Request и возврат Response. Никаких SQL-запросов или расчетов.
- Сервисы: вся бизнес-логика живет здесь.
- Репозитории/Models: работа с данными.

## Работа с данными и БД

- Наличие связей One-to-Many и Many-to-Many.
- Value Objects (VO)

Например: Аналог в [автомойке](README-task-car-wash.md): перевод рублей в копейки при сохранении и обратно при выводе. В вашем проекте: это может быть работа с весом, валютами, часовыми поясами или единицами измерения.

## Бизнес-логика

Проект не должен быть обычным CRUD-приложением. Должны быть реализованы:

Расчетные поля: наличие сущности, чьи параметры вычисляются на основе вложенных данных (например, общая сумма заказа, средний рейтинг, остаток на складе).

Валидация состояний (State Machine): сущность должна иметь статусы (например, «Новый» -> «В работе» -> «Завершен»). Переход между статусами должен накладывать ограничения на редактирование.

Бизнес-проверки: запрет на дублирование логических данных (например, нельзя добавить один и тот же товар в корзину дважды — нужно выкидывать ошибку с описанием).

По возможности можно показать реализацию optimistic lock

## Система ролей

Минимум 2-3 роли с разным уровнем доступа.

Например:

- Администратор: полный доступ к системе.
- Сотрудник: управление процессами, но без прав удаления/правки настроек.
- Клиент: просмотр только своих данных.

Т.е сотрудник может просмотреть только свои заказы. Клиент может посмотреть только свои заказы. Админ видит все.

Маршрут (api) у них один.

## Side-effect

Реализация триггеров: при определенном событии (например, смена статуса заказа) должна срабатывать логика отправки уведомления.

Почта: поставить через docker или отдельно через скрипт почтовый клиент (mailpit), продемонстрировать отправку почты.

## ДОПОЛНИТЕЛНО - по желанию.

Продемонстрировать событийную модель:

- заказ создан;
- заказ завершен;
- пользователь зарегистрирован;
- платеж подтвержден;
- сущность переведена в другой статус.

Асинхронная обработка и события

- события и listeners/handlers внутри приложения;
- брокер сообщений, например RabbitMQ;
- очереди для фоновых задач;
- паттерн Transactional Outbox для надежной публикации событий.

Примеры задач, которые можно выполнять асинхронно:

- отправка email;
- отправка уведомлений;
- интеграция с внешними сервисами;
- длительные фоновые операции.
